1. Project Overview

Smart Landmarks is a Flutter-based mobile application for managing geo-tagged landmarks through a REST API. Users can browse landmarks in map and list views, submit visits using live GPS location, add landmarks with image upload, and continue key actions in offline conditions through local persistence and delayed synchronization.

2. Features Implemented

  • Landmark retrieval from remote API and display in map and list interfaces.
  • Score-based sorting and minimum-score filtering support.
  • Interactive OpenStreetMap integration via flutter_map.
  • Visit workflow with device GPS capture and server-calculated distance.
  • Create landmark endpoint integration with multipart image upload.
  • Soft delete and restore operations for landmarks.
  • Local visit history tracking with time and distance.
  • SQLite cache for offline landmark access.
  • Pending visit queue for offline actions.
  • Automatic sync of pending actions after connectivity returns.

3. API Usage

Base URL: https://labs.anontech.info/cse489/exm3/api.php

Authentication: Query parameter key=24141210 in all requests.

  • GET action=get_landmarks: Retrieves all landmarks for map and list rendering.
  • POST action=visit_landmark: Submits user visit with JSON body containing landmark_id, user_lat, and user_lon.
  • POST action=create_landmark: Creates a landmark with title, lat, lon, and image using multipart/form-data.
  • POST action=delete_landmark: Soft-deletes a landmark by id using form-data.
  • POST action=restore_landmark: Restores a soft-deleted landmark by id using form-data.

4. Offline Strategy

  • Landmark data is cached in SQLite for offline browsing.
  • Offline visit actions are inserted into pending_visits queue.
  • Pending queue is reprocessed automatically on app startup or refresh when network is available.
  • Local visit history remains available even without connectivity.
  • Network state transitions are observed with connectivity_plus.

5. Architecture Used

  • State Layer: Provider with ChangeNotifier.
  • Service Layer: ApiService (network) and DatabaseService (SQLite).
  • Data Layer: Model classes for landmark, visit history, and pending visit.
  • Presentation Layer: Feature-based Flutter screens and widgets.
  • Separation of concerns between UI, state logic, API communication, and persistence.
  • Offline-first fallback design for unstable network environments.
  • Reusable services for maintainability and extension.

6. Challenges Faced

  • Managing multiple request body types across endpoints: JSON, form-data, and multipart/form-data.
  • Designing robust offline queue and replay logic without data loss.
  • Preventing duplicate visit submission during delayed synchronization.
  • Handling GPS permissions and location availability edge cases.
  • Keeping map marker color logic consistent with dynamic score values.
  • Safely parsing variable backend response structures.
